# MacroLog

Personal calorie/macro tracking PWA for Colin. Single user, single active device at a time. Hosted on GitHub Pages at `https://colinswhardy.github.io/claudemacro/` (repo `colinswhardy/claudemacro`). Backend is a single production Supabase project (Postgres + PostgREST + GoTrue auth) — there is no staging environment and no test database; every push to `main` deploys straight to the live site and the live DB.

## Files

- `index.html` (~3500 lines) — the entire app: HTML shell + `<style>` + one big inline `<script>`. No build step, no framework, no bundler, no TypeScript, no npm dependencies. A second tiny `<script>` at the bottom just registers the service worker.
- `sw.js` — service worker. Network-first for the app shell (cache name `macrolog-v2`), so a redeploy is picked up on next load instead of being stuck on a stale cache.
- `manifest.json` — PWA manifest.
- `supabase/schema.sql` + `supabase/README.md` — the DB schema and one-time setup walkthrough. Six tables, all RLS-scoped to `auth.uid() = user_id`: `food_log_entries`, `weight_entries`, `recipes`, `custom_barcodes`, `user_settings`, `user_app_data` (favorites/recentFoods/foodCache, one JSON blob per key).
- `tests/run-tests.js` — loads the real `<script>` block from `index.html` into a sandboxed Node `vm` context (with minimal `document`/`localStorage`/`history`/`fetch` stubs) and unit-tests the pure/state-only functions directly. Run with `node tests/run-tests.js` from the project root. Node is installed locally for this (`node --version` → v24+).

## Architecture

**Tiny hand-rolled render engine**, not a framework: every state change does a full `root.innerHTML = renderApp()` replace (see `render()`/`rerender()` near line 863, and don't skip reading the "Back button / gesture navigation" comment block right above it — it governs a lot of what "closing a screen" means throughout the file). Event handling is delegation on the root `#app` element via `data-action` (click), `data-bind` (text input, read on `input` event), `data-change` (file/select inputs), `data-enter` (Enter key). Three flat handler objects: `actions.X`, `inputActions.X`, `changeActions.X` — every `data-action="X"`/`data-bind="X"`/`data-change="X"` must resolve to a same-named entry in one of these, or it's dead on click.

**Critical ordering gotcha:** `actions`/`inputActions`/`changeActions` are declared via `const actions = {}` etc. partway through the file (~line 863 area). Any `actions.foo = ...` assignment placed *before* that declaration throws a TDZ `ReferenceError` that halts the entire script — the app just shows "Loading..." forever, on every device, with no console access on a phone to see why. This has happened before. Before any commit that adds new top-level `actions.X =` / `inputActions.X =` / `changeActions.X =` assignments, verify none of them appear above their object's `const` declaration.

**No build step = no compiler safety net.** A syntax error is a runtime failure on Colin's phone, not a red squiggle. Before every push: extract the two `<script>` blocks and run `node --check` on the big one, then `node tests/run-tests.js`. Both are fast (seconds) and are not optional — they're what catches the class of bug described above before it ships.

### Food tab screen model

`renderFoodTab()` gates on 7 mutually-exclusive flags on `state.ui.food` (via `foodUi()`), checked in priority order — first truthy one wins the whole screen: `scanningBarcode`, `loggingRecipe`, `recipeBuilder` (which itself nests a `picker` sub-state for the ingredient picker, and `picker.selecting`), `wizardType`, `showAI`, `manualEntry`, `addingFood`. Falls through to the search/history/favorites/recipes sub-tabs if none are set.

### Back-button / navigation (added 2026-07)

The app has no router. `computeNavDepth()` derives a 3-tier depth purely from `state` (0 = dashboard tab, 1 = any other tab at its base, 2 = any Food-tab sub-screen from the list above). `render()` calls `syncNavHistoryDepth()` at the top of every call, which auto-`pushState`s to match depth *increases* — no action needs to manually push history, entering any deeper screen is handled generically. Exiting a level is different: a screen's own Cancel/Back/Confirm-and-exit action must call `stepBackOneLevel()` instead of clearing its own flag directly, which triggers `history.back()`; the single `popstate` listener (bottom of file) is the one place that actually calls `collapseOneNavLevel()` and mutates state, so physical back/gesture and in-app buttons collapse identically. Actions that route through `stepBackOneLevel()` also lost the implicit double-submit guard they used to get from synchronously nulling their own flag — they carry an explicit `_confirmLock` 600ms debounce guard instead (grep `_confirmLock` for the pattern; apply it to any new confirm-and-exit action). A few transitions are deliberately *not* routed through `stepBackOneLevel()` because they're lateral swaps between two depth-2 screens, not a real exit (e.g. barcode-scan → adding-food after a successful lookup) — see the comment above `collapseOneNavLevel` for the full reasoning.

### Sync engine

Offline-first: `localStorage` is the instant source of truth; every save function also calls `scheduleCloudSync()` (debounced 2.5s) to push the full current dataset as a PostgREST upsert. `pullAndMergeAll()` runs on login and on boot and does everything in **pull → merge → push** order (settings first, then the rest). Merges are **in place** (never snapshot-and-replace-wholesale — that was a real bug), with newer-wins conflict resolution by `updated_at` for food logs and weights (recipes/customBarcodes stay additive-only). A "recently deleted" tombstone (`recentlyDeletedIds`, ~30s) stops a pull from resurrecting something just deleted locally. "Clear All Data" issues real `DELETE`s against all 6 tables — pushes are upsert-only and never delete.

**Why pull-before-push is load-bearing, not incidental (fixed 2026-07):** `favorites`/`recentFoods`/`foodCache` are each stored as one whole JSON blob in `user_app_data`, and their push is a **blind whole-blob overwrite** (additive merge, but overwriting push). If the push ran first, a device booting with an empty or truncated local store (fresh install, or iOS/PWA localStorage eviction) would overwrite the cloud master copy with its near-empty local one — silently wiping History for every device ("history is empty sometimes" — the actual reported bug). Pulling+merging first guarantees local already contains the cloud's data before any push, so the push always sends local-unioned-with-cloud, never a subset; it also makes food-log newer-wins correct (the blind id-upsert can't clobber a newer cloud edit). `pullAndMergeAll` also **bails before pushing if any pull errored** — pushing on top of a failed pull is exactly the clobber risk; the debounced per-save sync still covers un-pushed local changes. Do not reorder this back to push-first, and do not add a new whole-blob collection without keeping it inside this pull-first flow.

**Debounce vs. backgrounding (fixed 2026-07)**: the 2.5s debounce was a real data-loss window — logging food and immediately backgrounding/locking the phone (completely normal usage) could kill the pending push before it fired, leaving that entry stranded in local-only storage on one device with no cloud copy. Fixed two ways: `scheduleCloudSync` now tracks the pending push function in `pendingSyncPushFns` alongside its timer, and `flushPendingCloudSyncs()` (wired to both `visibilitychange`→hidden and `pagehide`) fires any still-pending push immediately instead of waiting out the timer; `supabaseTable`'s fetch also sets `keepalive: true` so an in-flight request survives the page actually being torn down, not just backgrounded. Any new debounced-write pattern should route through `scheduleCloudSync` rather than rolling its own timer, or it won't get this protection.

**New field on a table-backed row (not a whole-blob collection) needs a real column, and check before shipping.** `food_log_entries` is a fixed-schema table — adding a JS field like `entry.brand` and just including it in `flattenFoodLogsForSync`'s payload before the column exists would make PostgREST reject the *entire* push (unknown column), silently breaking sync for that whole table until fixed. `entry.brand` and `entry.animalOverride` (added 2026-07) currently ship **local-only** — captured on the entry, shown/editable in the UI, but deliberately left out of `flattenFoodLogsForSync`/the pull `select=`/`mergeFoodLogsFromCloud` — specifically to avoid this. `supabase/README.md` has the `alter table` migration to run; once it's confirmed run, wire these two fields into the sync functions (mirroring the existing calories/protein/etc. fields) and mention the change needs `schema.sql` updated too (already done) plus a fresh install to pick it up automatically. Don't add sync payload fields for columns that may not exist yet — verify against `supabase/schema.sql` (or ask) first.

## Conventions

- Text inputs deliberately do **not** trigger `rerender()` on keystroke (would steal focus mid-type) — `inputActions.X` handlers patch specific DOM elements directly (e.g. flipping a button's `disabled`) instead of re-rendering. This is a recurring source of "button doesn't update" bugs when adding a new input; check for this pattern before assuming a fresh flag will "just work."
- **Serving-size quantity selection** (added 2026-07): the three "pick a food, then set how much" screens (Adding Food, recipe ingredient picker, wizard result) share `renderServingControls`/`computeQtyWeight`/`patchMacroPreview` (near `macroStat`, ~line 2338). Tapping a serving chip (e.g. "1 cookie (33g)") sets a `servingBasis` and switches the input to a quantity ("2" → 66g) instead of raw grams; "Enter grams instead" clears `servingBasis` to go back to typing grams directly. Each screen keeps its own `weight`/`servingBasis`/`qty` trio in its own UI sub-object (`fu.addWeight`/`addServingBasis`/`addQty`, `pk.weight`/`servingBasis`/`qty`, `wiz.weight`/`servingBasis`/`qty`) — all must be reset together (`servingBasis = null; qty = "1"`) at every point a new food is selected/estimated, or a stale quantity basis will carry over to an unrelated food.
- **Auto-calculated calories** (added 2026-07): Manual Entry and the "barcode not found" custom-entry screen both derive calories from protein/carbs/fat via `calcCaloriesFromMacros` (Atwater 4/4/9, same factors as the Strategy tab's target math) — editing any of the three macro fields recomputes and DOM-patches the calories input. Editing calories directly still works and isn't overwritten unless a macro field is touched again afterward (it's a one-way derivation, not a two-way lock).
- **Search-as-you-type, two separate groups** (2026-07): the Search sub-tab renders **two independent lists**, patched into `#foodResultsBlock` via `patchFoodResultsBlock()` rather than a full `rerender()` (which would blur the input mid-keystroke). `fu.historyResults` = "★ Eaten before" (from `localFoodMatches(q)`: a synchronous substring match over `recentFoods`/`favorites`/`foodCache`, no network, fires from the **1st** character) renders on top; `fu.results` = "Food databases" (USDA + Open Food Facts via debounced `liveFoodSearch(q, mySeq)`, 450ms after the last keystroke, 3+ chars to spare USDA's shared rate-limited demo key) renders underneath. **Keep these two lists separate** — they used to be concatenated through `deduplicateFoods` together, which folded an already-eaten food into a database row and destroyed its "you've had this before" identity. `dbResultsExcludingHistory(history, dbFoods)` de-dupes USDA-vs-OFF *within* the database group and drops anything already shown in the history group by id, so nothing appears twice. Both this and the explicit tap/Enter path (`doFoodSearch`) share one module-level `foodSearchSeq` counter bumped on every keystroke/explicit search; an in-flight request checks its captured `mySeq` before applying results, so a slow older request can't clobber a newer one — preserve that guard. `selectFoodForAdd` must search `fu.historyResults` too (those rows can originate from favorites/foodCache, not just `recentFoods`).
- **Plant-protein estimate** (added 2026-07): the dashboard's Protein bar shows a "(Xg plant-based)" sublabel, same spot fiber annotates the Carbs bar. Unlike fiber, no database provides this — `isAnimalProteinFood(name)` (near `getDayTotals`, ~line 1214) is a keyword regex guess against the logged entry's name (meat/dairy/egg/fish all count as animal, per Colin's call — eggs and fish aren't literally meat or dairy but count the same way). It's computed live in `getDayTotals`, not stored or synced, so it applies retroactively to old entries and needed no schema change — but it's a heuristic: a mixed dish matching any animal keyword is attributed entirely to animal protein (floor, not exact), and an unrecognized packaged/branded name with real animal protein could be missed. Keep new keywords `\b`-word-boundary-wrapped (`egg` must not match inside `eggplant`) and add solid compound words explicitly when boundary matching can't catch them (`milkshake`, `cheeseburger`, `hamburger` — no boundary exists between the two halves).
- **Adaptive calorie recommendations** (added 2026-07): the Strategy tab's Weight Goals section shows a "Recommended Calories" block driven by two pure-ish functions near `renderWeightGoalsSection` (~line 3719): `estimateMaintenanceCalories()` measures TDEE from Colin's own last 28 days (average intake on plausibly-complete food days — ≥800 kcal — minus the energy equivalent of the actual weight change; 3500 kcal/lb or 7700 kcal/kg via `KCAL_PER_WEIGHT_UNIT`; returns null below 10 food days / two weigh-ins spanning 14+ days), and `computeGoalPlan(maintenance)` turns that into by-target-date and by-weekly-rate daily-calorie recommendations (signed `remaining` makes cuts land below maintenance and bulks above, automatically). New setting `goalRatePerWeek` (magnitude; direction inferred from goal vs. current). The block live-updates via `patchGoalPlanBlock()` from the goalWeight/goalDate/goalRatePerWeek `inputActions` — the standard no-rerender-on-keystroke pattern. Recommendations under 1200 kcal render with an "aggressive" warning rather than being clamped.
- Numeric parses use `Math.max(0, parseFloat(x) || 0)` to reject negatives; falsy-zero-prone spots use an explicit `isNaN()` check instead of `|| default` where `0` is a legitimate value (e.g. weight).
- Double-submit guards on "confirm and add" actions use a `_confirmLock`/`_aiAddLock`-style timestamp debounce (600ms) stored on the relevant UI sub-object, not a DOM `disabled` flip alone.
- **Two-step delete confirmation** (added 2026-07): recipes, dashboard log entries, weight entries, and custom barcodes all use an arm/confirm/cancel pattern instead of deleting on a single tap — a UI field holds the id/date/code pending confirmation (`fu.confirmDeleteRecipeId`, `state.ui.confirmRemoveEntryId`, `wu.confirmDeleteDate`, `su.confirmDeleteBarcodeCode`), the row conditionally renders a Cancel/"Delete?" pair instead of its normal buttons when it matches, and only the confirmed action actually deletes (and clears the flag). Follow this same shape for any new destructive single-tap action rather than adding a bare confirm() dialog or deleting immediately.
- **Deleting a row that syncs to Supabase needs three things, not just a local delete**: (1) remove it from local state, (2) `scheduleCloudDelete(table, idColumn, idValue)` to issue the real DELETE (push functions are upsert-only and never remove cloud rows on their own), (3) make sure that table has a bucket in `recentlyDeletedIds` (near `mergeFoodLogsFromCloud`) and that its `merge*FromCloud` function checks `wasRecentlyDeleted(table, id)` before re-adding a row — otherwise a background pull racing the DELETE resurrects what was just removed. `weight_entries` (keyed by `log_date`, no local `id` tracked) and `custom_barcodes` (keyed by `barcode`) both went through this in 2026-07 alongside `food_log_entries`/`recipes`.
- **Copy a log entry to another date** (added 2026-07): a 4th "Copy" tab in the log-entry editor (`renderEntryEditor`, alongside Portion/Macros/Move Day), backed by `copyFoodEntryToDate` right next to `moveFoodEntryToDate`. Deliberately different from move in two ways: copying onto the *same* date the entry is already on is allowed (a legitimate "log a second serving" case, not a no-op), and the actions (`copyEntryYesterday`/`copyEntryTomorrow`/`copyEntryCustomDate`) don't call `closeEntryEditor()` — they `showToast()` and leave the editor open so copying the same item to several dates doesn't require reopening it each time. Both Move Day and Copy also have a "Today" quick button (`moveEntryToday`/`copyEntryToday`) alongside Yesterday/Tomorrow, for jumping an entry back to the real current date while viewing an older day's log — Yesterday/Tomorrow are relative to whatever day is being viewed, not to "today," so they don't cover that case on their own.
- Recipe deletion, log-entry removal, weight-entry deletion, and custom-barcode CRUD all have direct unit test coverage in `tests/run-tests.js` (arm-but-don't-delete, confirm-actually-deletes, tombstone-blocks-resurrection) — extend these rather than only hand-testing on the phone when touching this code.
- **History = durable "everything ever added"** (2026-07): the Food tab's History sub-tab is `state.recentFoods`, recency-ordered and deduped by food id, meant as a lasting searchable list to re-add past foods (not a per-day log — the dashboard's day nav is that). Cap is **500** (was 50) in `addFoodToLog`. Every logging path (search/barcode/manual/recipe/AI) already funnels through `addFoodToLog`, so all sources land here automatically — don't add a logging path that bypasses it.
- **Barcode corrections for future scans** (2026-07): tapping a barcode-scanned log entry (Portion/Macros tabs) shows "Save correction for future scans" → `saveBarcodeCorrection` snapshots the entry's current weight+macros into `state.customBarcodes[code]` (keyed via `barcodeCodeForEntry`, which only matches real barcode sources — `barcode_*` foodIds and `"Open Food Facts (Barcode)"`, deliberately not text-searched OFF results). `handleBarcodeDetected` checks `customBarcodes[code]` before hitting OFF, so the next scan uses the correction; it also appears in Strategy > Data > Custom Barcodes. Stored per-serving (`per100g:false`, one `servingSizes` entry) like a hand-entered barcode.
- **Weight chart X axis is time-proportional** (2026-07): `weightChartXPositions(dates, x0, plotWidth)` (pure, tested) places each weigh-in by real elapsed time from the first date, so a skipped day is a real gap — not evenly-spaced-by-index. `renderWeightChart` uses it. Keep any new time-series chart on this helper rather than index spacing.
- **`calcMacrosForWeight`'s base-grams bug (fixed 2026-07): what portion do a food's stored macros describe?** Database foods (USDA/OFF) are per-100g (`per100g:true`). Everything the user *creates* (Manual Entry, AI Estimate, Recipe, custom barcodes) stores **absolute** macros for one specific portion instead — the portion is recovered via `foodBaseGrams(food)`: `per100g` → 100, else `servingSizes[0].grams`, else `loggedWeight`, else 100 as a last resort. This was a real reported bug ("Loonie Dog" logged at 235 kcal/83g came back as 195 kcal when re-added from History): absolute-portion foods had no `servingSizes`, so the old code assumed a 100g base and rescaled 235 × (83/100). `foodQuantityLabel(food)` mirrors this for the "kcal/Xg" text next to a search result — never hardcode "kcal/100g" for a food that might not be per-100g. Any new logging path that stores absolute macros **must** set either `servingSizes` or `loggedWeight` (ideally both) on the food object, or re-adding it later will silently rescale.
- **Editing a logged entry, "everything" (2026-07)**: the entry editor (`renderEntryEditor`) has 5 tabs now — Portion, Macros, **Details** (new: name, brand, per-entry plant/animal protein override via `saveEntryDetails`), Move Day, Copy. `entryBrandLine(entry)` suppresses the brand line on the dashboard card when it would just repeat the source line underneath (e.g. brand "Manual Entry" == source "Manual Entry") — real, distinct brands (USDA/OFF, or a manual correction) still show. `entryHasZeroMacros(entry)` flags a card with a ⚠ warning when protein/carbs/fat are *all* exactly 0 (deliberately not calories — a food can validly have real calories with 0g of all three, e.g. alcohol).
- **Plant-protein override**: `isAnimalProteinEntry(entry)` checks `entry.animalOverride` (`null`=Auto/keyword-guess, `true`/`false`=manual) before falling back to `isAnimalProteinFood(entry.name)` — `getDayTotals` calls this, not the raw keyword function, when computing the plant-protein rollup. New setting `showPlantProtein` (Strategy > Preferences) gates whether the dashboard sublabel shows at all.

## Deploy

**Before every commit that ships a real change**, bump `APP_VERSION` (near the top of `index.html`, ~line 170) up by `0.01`. It's shown at the bottom of Strategy > Data ("MacroLog vX.XX") specifically so Colin can glance at his phone and tell whether a push actually landed, instead of guessing against a possibly-stale service-worker cache — an unbumped version number after a push silently defeats the entire point of it. Started at `1.0` (2026-07); do not skip this step, and do not bump it for something that isn't actually shipping (e.g. a revert-only commit, a pure `.md` edit).

`git push origin main` → GitHub Pages auto-builds, typically live within 1–2 minutes. No CI, no staging slot. Verify a push landed with a direct fetch, e.g.:
```powershell
(Invoke-WebRequest -Uri "https://colinswhardy.github.io/claudemacro/index.html" -Headers @{"Cache-Control"="no-cache"}).Content -match "some-string-from-the-change"
```

## Known deferred items (raised, explicitly not implemented)

- API keys (Claude/Gemini) sync to the cloud as part of the settings blob — a UX tradeoff (cross-device convenience) left as-is, not an oversight.
- USDA search uses the shared `DEMO_KEY` — needs Colin to register a free personal key at api.data.gov. Live search-as-you-type (see Conventions) increases call volume against this shared key somewhat; the 450ms debounce + 3-char minimum is the mitigation for now.
- Recipes' merge conflict resolution is additive-only (no `updated_at` column on that table) — upgrading it needs a schema migration first.
- Feature ideas from a competitive review (activity logging, meal templates, water tracking, micronutrients, label OCR, etc.) were explicitly scoped out as reference-only, not a backlog.
